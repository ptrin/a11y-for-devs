Building empathy

Diverse abilities

  • People have diverse abilities, skills, tools, preferences, and expectations that can impact how they use the Web
  • Many reasons why people may experience varying degrees of auditory, cognitive, physical, speech, and visual disabilities
  • May have disabilities from birth, an illness, disease, or accident, or develop impairments with age
Source: W3C Web Accessibility Initiative: Diverse Abilities and Barriers

The ability spectrum

By designing for someone with a permanent disability, someone with a situational limitation can also benefit.
For example, a device designed for a person who has one arm could be used just as effectively by a person with a temporary wrist injury or a new parent holding an infant.

Source: Microsoft Inclusive Design Toolkit

Models of disability

  • Medical
    • Individual level, sees disability as deficient/defective aspect
  • Social
    • Sees disability as part of diverse experience of humanity which can be accounted for by society and technology

Social model of disability

  • Identifies systemic barriers, negative attitudes and exclusion by society (purposely or inadvertently)
  • Individual limitations or impairments do not have to lead to disability
  • Social model focuses on changes required in society
  • People can be disabled by a lack of resources to meet their needs
Source: Wikipedia - Social model of disability
Source: What responsive design can teach us about accessibility

Impact of web accessibility

  • The Web is part of the solution to improving access to information and services for people with disabilities, as well as overall equity in society
  • An accessible Web expands opportunities for communication, social interaction, and community participation for people with disabilities and people with age-related or other impairments.
  • The technology exists to remove online barriers – WCAG, other web standards, vendor software built to W3C specifications
Source: What responsive design can teach us about accessibility

Barriers on the web

  • Images, controls, and other structural elements that do not have equivalent text alternatives
  • Not providing full keyboard support
  • Unpredictable focus handling
  • Low contrast
  • Relying on colour or position to communicate information

Accessibility fundamentals

W3C Web Accessibility Initiative

  • Accessibility is a first-class citizen of W3C web standards
  • Accessibility Initiative produces specifications for web content, user agents, and authoring tools

WCAG Principles

Anyone who wants to use the Web must have content that is:

  • Perceivable
    Information and user interface components must be presentable to users in ways they can perceive.
  • Operable
    User interface components and navigation must be operable.

WCAG Principles

  • Understandable
    Information and the operation of user interface must be understandable.
  • Robust
    Content must be robust enough that it can be interpreted reliably by a wide variety of user agents and AT.

Perceivable

  • Provide text alternatives for non-text content.
  • Provide captions and other alternatives for multimedia.
  • Create content that can be presented in different ways, including by assistive technologies, without losing meaning.
  • Make it easier for users to see and hear content.
Source: WCAG 2.1 at a Glance

Operable

  • Make all functionality available from a keyboard.
  • Give users enough time to read and use content.
  • Do not use content that causes seizures.
  • Help users navigate and find content.
Source: WCAG 2.1 at a Glance

Understandable

  • Make text readable and understandable.
  • Make content appear and operate in predictable ways.
  • Help users avoid and correct mistakes.
Source: WCAG 2.1 at a Glance

Robust

  • Maximize compatibility with current and future user tools.
Source: WCAG 2.1 at a Glance

Development habits for accessibility

Always be testing!

  • Demo: Axe and WAVE browser extensions

Accessibility tree

  • Browser accessibility tree is a subset of the DOM
  • Browser uses defined algorithms to determine accessible name and properties/state of element
  • Browser communicates accessibility tree to the OS accessibility API
Source: Assistive Technology and the Accessibility Tree

Writing accessible code

  • Make it keyboardable
  • Use semantic HTML
  • Ensure all content is perceivable

Make it keyboardable

  • Ensure that you can tell which element has keyboard focus
  • Keyboard navigation order should be logical and intuitive
  • All functionality must be available to the keyboard
    • Following ARIA interaction guides if applicable
  • Users should be able to tab into and out of any UI element
    • With the exception of intentional tab trapping for modal dialogs

Use semantic HTML

  • All structure and informative content is provided through HTML
  • Semantic HTML and landmark roles are used to structure HTML elements
    • Used by screen readers for navigation
  • Elements that visually look like headings are marked up as such
  • Ensure that images have alt text, except decorative images which should have role="presentation" and empty alt text

Ensure all content is perceivable

  • Programmatic associations are provided for form controls and labels
  • Invalid fields need to use aria-invalid
  • Required fields need to use aria-required
  • Information & instructions are programmatically conveyed to AT
    • e.g. using aria-describedby
  • Dynamic page updates need to be communicated to screen readers
    • e.g. focus is set to the error message or summary when errors are returned

What is ARIA?

  • Accessible Rich Internet Applications standard
  • Extends semantics beyond what is possible using native HTML
  • Allows content authors to modify the accessibility tree to augment and override native semantics
  • Provides screen reader users with more context and greater interactivity with content
  • Uses HTML attributes either directly in markup or added via JavaScript

What is ARIA?

  • ARIA has no effect on how elements are displayed or behave in browsers
    • e.g. aria-expanded true/false does not expand/collapse elements, JavaScript and CSS need to do that
  • It doesn’t add functionality, only metadata about DOM elements

ARIA roles

  • Provides information to AT about the identity of an element
  • Roles also determine which states and properties are allowed/valid
  • Types of role:
    • Landmark e.g banner, contentinfo, navigation, search
    • Widget e.g. tablist, tab, tabpanel, progressbar, slider, tooltip
    • Document e.g. presentation, region, dialog
  • A role is a promise that content will behave according to ARIA guidelines

ARIA properties

  • Often describe relationships with other elements
  • Usually don't change once they're set

Examples:

  • aria-label
  • aria-labelledby
  • aria-describedby
  • aria-controls
Full list: https://www.w3.org/TR/aria-state/#state_prop_def

ARIA states

  • Dynamic, typically updated with JavaScript
  • Reflect current values / state related to element

Examples:

  • aria-invalid
  • aria-selected
  • aria-expanded
  • aria-valuenow

Using ARIA

  • If you can use a native HTML element, then do so
  • Do not change native semantics e.g. <h2 role="tab">
  • All interactive ARIA controls must be usable with the keyboard e.g. role="button"
  • Do not use role="presentation" or aria-hidden="true" on a visible, focusable element
  • All interactive elements must have an "accessible name"
Source: https://www.w3.org/TR/aria-in-html/#notes2
